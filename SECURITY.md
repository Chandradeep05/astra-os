# Security Policy

We take the security and safety of ASTRA-OS very seriously. Since ASTRA-OS is a local-first runtime capable of running arbitrary code inside a sandbox and interacting with your local filesystem, security boundaries are a top priority.

## Supported Versions

Currently, security updates are provided for the active release line:

| Version | Supported |
| ------- | --------- |
| v0.4.x  | ✅ Yes    |
| < v0.4  | ❌ No     |

---

## 🔒 Security Design Principles

ASTRA-OS operates under strict security boundaries:
1. **Local-Only Boundary**: All network operations are strictly local. No telemetry, third-party trackers, or cloud storage syncing exists.
2. **Restricted Execution**: Any Python code execution generated by the agent is locked inside a `RestrictedPython` sandbox with AST inspection to block direct module imports (like `subprocess`, `os`, `sys`) and limit cpu/memory usage.
3. **Human-in-the-Loop Gate**: All risky tool calls (file writes, execution requests, system modifications) are blocked by default and require explicit user approval.
4. **PII Redaction**: Personal identifying information is sanitized at three layers before being persisted in episodic memory.

---

## ⚠️ Reporting a Vulnerability

If you discover a security vulnerability (such as a sandbox escape, a way to bypass the approval gate, or auth bypasses on local API routes), **please do not open a public issue.** 

To report a vulnerability privately:
1. Contact the maintainers directly or via email at the address associated with this repository.
2. Include a detailed description of the vulnerability, steps to reproduce, and a Proof of Concept (PoC) if available.
3. We will respond within 48 hours to confirm receipt and discuss a resolution timeline.

Thank you for helping keep ASTRA-OS secure for everyone!
